# MiniKafka (Go) — 7일 실행 계획

## AI 사용 규칙 (이 프로젝트의 핵심 제약)

- ❌ **금지:** 기능 코드 생성 요청 ("~구현해줘"), 파일 통째로 작성
- ✅ **허용:** 개념 질문, 내가 짠 코드 리뷰, 에러 메시지 디버깅 힌트, Go 관용구 확인
- 규칙: 막히면 최소 30분은 스스로 부딪힌 뒤에 질문한다. 질문할 땐 "정답"이 아니라 "방향"을 요구한다.
- 이유: AI가 짜면 하루면 끝나지만 아무것도 남지 않는다. 이 프로젝트의 산출물은 코드가 아니라 **내 머릿속 모델**이다.

## Day 1 — Go 발 담그기 + TCP 왕복

**만들 것**

- Go 프로젝트 구조 (`cmd/broker`, `internal/...`), go mod 세팅
- `net` 패키지로 TCP 서버: 클라이언트가 보낸 바이트를 그대로 돌려주는 에코 서버
- goroutine으로 커넥션별 핸들링 (`go handleConn(conn)`)

**완성 기준**

- `nc localhost 9092`로 접속해서 친 글자가 돌아온다
- 커넥션 2개 동시에 붙여도 각각 독립적으로 동작한다

**배우는 것:** goroutine 동시성 모델, Go 에러 처리 관습, defer

## Day 2 — Append-Only 로그 (심장)

**만들 것**

- `internal/log`: 메시지를 파일에 append하는 로그 구조체
- 바이너리 인코딩: `[4바이트 length][payload]` length-prefix 방식
- offset(N번째 메시지) 기준으로 읽기 — 처음엔 순차 스캔으로 OK
- 인덱스 파일(offset → 파일 위치)은 스캔이 느려지는 걸 몸으로 느낀 **뒤에** 추가

**완성 기준**

- `log.Append([]byte)` → offset 반환
- `log.Read(offset)` → 해당 메시지 반환
- 프로세스 재시작 후에도 데이터가 살아있다 (복구 로직)

**배우는 것:** 왜 로그는 append-only인가, 바이너리 직렬화, `encoding/binary`

## Day 3 — 프로토콜: Produce / Fetch

**만들 것**

- 커스텀 바이너리 프로토콜 설계: `PRODUCE(topic, payload)` / `FETCH(topic, offset)`
- Day 1의 TCP 서버 + Day 2의 로그를 연결
- 얇은 Go 클라이언트 라이브러리 (`client.Produce()`, `client.Fetch()`)

**완성 기준**

- 클라이언트 A가 produce → 클라이언트 B가 offset 0부터 전부 fetch
- 컨슈머가 자기 offset을 스스로 관리한다 (브로커는 누가 어디까지 읽었는지 모름)

**배우는 것:** 카프카의 핵심 설계 — **"브로커는 멍청하게, 컨슈머는 똑똑하게"**가 왜 확장성을 만드는지

---

## Day 4 — 파티셔닝 (순서 vs 처리량의 화해)

**만들 것**

- 토픽 = 파티션 N개, 파티션 = 독립된 로그
- 키 기반 라우팅: `hash(key) % numPartitions`
- 파티션별 디렉토리 구조 (`data/{topic}/{partition}/`)

**완성 기준**

- 같은 키의 메시지는 항상 같은 파티션 → 키 단위 순서 보장 확인
- 다른 키들은 병렬 파티션에 분산되는 걸 확인

**배우는 것:** 전체 순서는 포기하고 키 단위 순서만 지키는 타협이 왜 대량 처리를 가능하게 하는가 — **이 프로젝트에서 제일 값진 하루**

---

## Day 5 — 컨슈머 그룹 (미니 버전)

**만들 것**

- 그룹 내 컨슈머들에게 파티션 정적 할당 (컨슈머 2개, 파티션 4개 → 2개씩)
- offset 커밋: 브로커에 `__offsets` 저장 (파일 기반이면 충분)

**⚠️ 스코프 컷:** 동적 리밸런싱(컨슈머 죽으면 재할당)은 **v2로 명시적으로 버린다.** 이거 시작하면 일주일 더 걸린다.

**완성 기준**

- 컨슈머 2개가 파티션 나눠서 병렬 소비, 재시작 시 커밋된 offset부터 이어 읽기

**시간 없으면:** 이 날을 통째로 버리고 Day 6·7을 지킨다. 컨슈머 그룹 없어도 프로젝트는 성립한다.

## Day 6 — 부하생성기 + fsync 실험 (숫자가 태어나는 날)

**만들 것**

- CLI 부하생성기: `./loadgen --events 100000 --rate 5000 --keys 100`
- 처리량(msg/s), 지연(p50/p99) 측정
- **fsync 정책 실험:** ① 매 write마다 fsync ② N건마다 배치 ③ OS에 위임 — 세 가지 처리량 비교

**완성 기준**

- 십만 건 쏘고 유실 없이 전부 소비되는 걸 확인
- fsync 정책별 처리량 차이를 **표로 기록** (이게 이력서 재료)

**배우는 것:** 내구성 vs 처리량 트레이드오프를 숫자로 체감 — 카프카가 왜 페이지 캐시에 기대는지
