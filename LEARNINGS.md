## [2026-07-07] Go 동시성 모델 — Node 멘탈 모델을 잘못 얹음

- **상황:** Day 1 시작 전 goroutine/스레드 개념 잡는 중
- **증상/오해:** "OS 스레드 100개면 동시에 active인 goroutine도 100개"라고 생각. 또 goroutine이 특정 스레드에 종속돼 못 옮겨간다고 오해. Read 이후 로직이 Node처럼 메인으로 "돌아와 끼어든다"고 생각.
- **원인:** Node의 "스레드 1개 + 이벤트 루프 끼어들기" 모델을 Go에 그대로 투영함.
- **배운 것:**
  - 동시에 진짜 도는 goroutine 수 = **코어 수(GOMAXPROCS)**, 스레드 총 개수 아님.
  - goroutine은 스레드에 종속 안 됨 — park됐다 다른 스레드에서 재개(work-stealing).
  - Read 다음 줄은 **같은 goroutine, 같은 직선 흐름**에서 이어짐. 콜백/메인 복귀 없음.
  - I/O 대기 goroutine은 스레드 0개 점유(netpoller) + 메모리 ~2KB만.
  - 순수 I/O 처리량은 Node ≈ Go. Go의 진짜 우위는 **여러 코어 공짜 + 블로킹 스타일 코드**.

## [2026-07-07] 메모리 공유 — Node worker와 정반대 트레이드오프

- **상황:** worker_threads의 shared-nothing 문제를 Go도 겪는지 질문
- **증상/오해:** "Go는 자원 공유 문제가 없다"로 끝날 뻔함
- **원인:** 공유의 대가(데이터 레이스)를 아직 안 봄
- **배운 것:** Go goroutine은 같은 힙 공유 → 포인터로 넘김(복사·직렬화 0). 대신 **데이터 레이스**가 열림 → `sync.Mutex` 또는 channel로 동기화는 내 책임. `go run -race`로 탐지. → **여러 producer가 같은 파티션 로그에 append할 때 정면으로 걸린다(Day 2~3).**

## [2026-07-07] Day 1 — Read 에러 시 루프 안 빠지면 goroutine이 무한 스핀

- **상황:** 에코 서버 `handleConn`에서 클라가 끊긴 뒤 서버 터미널이 EOF 로그로 도배됨
- **증상/오해:** `conn.Read`가 에러(EOF)를 줘도 출력만 하고 `for` 루프를 계속 돌게 짬 → 죽은 소켓에 Read 반복 → goroutine이 CPU 100% 태우며 영원히 삼
- **원인:** "에러를 값으로 다룬다 = 출력만 하면 끝"이 아님. 에러에 **행동**(루프 종료)을 안 붙임
- **배운 것:**
  - `Read`가 에러 = 이 커넥션 끝. `break`로 루프 탈출 → 함수 리턴 → **goroutine 정상 종료.**
  - goroutine은 자기 일 끝나면 **스스로 종료 조건**을 만나야 한다. 안 그러면 유령 goroutine leak → 오래 켜두는 브로커엔 치명적.
  - EOF는 "실패"가 아니라 정상 종료 신호 — 에러/정상을 구분해서 다룰 것.
- **남은 폴리시(미적용):** ①EOF일 때 "closed"와 "Failed" 둘 다 찍힘(else로 분기 필요) ②`defer conn.Close()` 없음 ③`main`의 listen 에러 시 `return` 안 해서 nil `ls.Accept()` panic 위험. → 셋 다 이후 커밋에서 수정 완료.

## [2026-07-07] defer는 "실행 횟수"만큼 예약이 쌓인다

- **상황:** `defer conn.Close()`를 에러 브랜치 안(break 직전)에 뒀다가 함수 맨 위로 옮김
- **증상/오해:** "for 밖이면 아무데나 OK, for 안 if 전이면 큰 문제 없다"고 생각
- **원인:** defer를 "블록 끝나면 실행"으로 오해. 실제론 **함수 리턴 시** 실행되고, **defer 줄이 실행될 때마다 예약이 1개씩 쌓임**
- **배운 것:**
  - defer는 함수가 **어느 경로로 나가든** 예약된 걸 실행 → 그래서 **자원 획득 직후(함수 입구)** 에 두면 모든 출구를 자동 커버.
  - 브랜치 안/유일한 출구 앞에 두면 = 그냥 직접 호출한 거랑 같음. 나중에 출구(return) 늘면 그 경로는 정리 누락 → fd 누수.
  - **for 안 defer = 반복마다 예약 누적.** conn.Close는 에러 무시라 우연히 안 터지지만, `mu.Unlock()`이면 이중 unlock으로 **panic**. → for 밖에 두는 건 취향이 아니라 정확성.

## [2026-07-07] Read 버퍼는 내가 소유, `recv[:n]`은 복사가 아니라 뷰

- **상황:** `recv := make([]byte, 4096)` 의미 파악
- **배운 것:**
  - `conn.Read`는 메모리를 스스로 안 만든다. **내가 준 버퍼에 채우고 `n`(채운 양)만 반환.** 버퍼는 내가 소유.
  - 버퍼는 for 밖에서 한 번 만들어 **재사용** → 매 Read가 이전 내용을 덮어씀.
  - `recv[:n]`은 새 메모리가 아니라 **버퍼를 가리키는 뷰(창문)**. 바로 쓰면 안전, 오래 들고 있으면 다음 Read가 덮어써서 깨짐.
  - `string(recv[:n])`은 **복사**를 하므로 안전. → **Day 2 복선:** 메시지를 저장하려면 `append(msgs, recv[:n])` 하지 말고 **copy해서** 들고 있어야 함("창문이지 사진이 아니다").

## [2026-07-07] offset은 물리 위치가 아니라 논리 ID + 세그먼트/상대오프셋 설계

- **상황:** Day 2 개념 — 로그가 무한히 커지고 스캔이 느려지는데 진짜 카프카는 어떻게 사나
- **오해했던 것:** ①flush = 오래된 데이터 밀어내기 ②앞 데이터 지우면 offset이 밀린다 ③번호가 무한히 커진다
- **배운 것:**
  - **flush(fsync, 내구성) ≠ retention(삭제).** flush는 데이터·offset 안 건드림. 삭제는 리텐션(시간/용량 정책)이 따로 함.
  - **offset은 배열 인덱스가 아니라 auto-increment PK.** 논리 일련번호라 앞을 지워도 재번호 안 됨. `id=3` 지워도 `id=6`은 그대로 6. → "offset은 영원한 주소"와 "옛날 데이터 삭제"가 모순 없이 공존.
  - **삭제는 세그먼트(1GB 조각) 단위로 파일 통째 `rm`** — 중간 파내기 아님.
  - **글로벌 offset은 int64(~920경)라 리셋 불필요.** 초당 100만 건 써도 29만 년치. "무한 확장"은 실무적 비문제.
  - **저장 효율은 (세그먼트 base offset + 상대 offset)으로 분해.** 세그먼트 파일명 = base offset, 인덱스는 상대 offset(4byte)만 저장. 글로벌 = base + 상대. 조회는 2단(세그먼트 이진탐색 → 세그먼트 내 인덱스 조회).
  - 순진한 로그(파일 하나·무한 append·처음부터 스캔)는 **작을 땐 완벽, 크면 죽음.** 살리는 3장치 = **리텐션+세그먼트 / length-prefix / 인덱스.**
- **v1 스코프:** length-prefix만 지금 구현. 인덱스는 스캔 병목 느낀 뒤 옵션. 세그먼트/리텐션은 v2(개념만).

## [2026-07-07] Go의 `*`(포인터) — struct는 기본 "값"이라 복사된다 (JS와 반대)

- **상황:** struct에 `*os.File`을 담으며 `*`가 왜 붙는지 궁금
- **핵심 차이:** JS/TS는 객체가 **기본 참조(reference)** — `b = a`면 같은 객체 공유. Go는 struct가 **기본 값(value)** — `b = a`면 **복사본**이 생김.
- **배운 것:**
  - 파일 핸들처럼 상태(fd, 현재 위치)를 든 건 복사되면 재앙 → `os.OpenFile`은 값이 아니라 **`*os.File`(포인터)** 반환. 모두가 같은 파일 하나를 공유.
  - `*` = "복사본 말고 진짜 그거 하나를 공유". `&`는 짝 — "이 값의 포인터를 가져와"(`&Log{...}`).
  - 같은 이유로 `func (l *Log)`(리시버), `NewLog() (*Log, error)`(반환)도 포인터. 값이면 복사본이라 상태 유지·공유 안 됨.
  - **규칙:** 상태 있고 공유돼야 하는 것(파일·커넥션·Log) → 포인터(`*`). 단순 값(offset 같은 숫자) → 포인터 안 씀. 그래서 `Append` 반환 `int64`엔 `*` 없음.

## [2026-07-07] `:=` vs `=` — 에러 변수는 재사용한다

- **상황:** `Append`에서 두 번째 에러를 `err2`로 새로 선언했다가 교정
- **배운 것:**
  - `:=` = 새 변수 선언+대입, 왼쪽에 **새 변수 최소 1개** 필수. `=` = 기존 변수 재대입.
  - `err`가 이미 있으면 `_, err = f()`로 **재사용**(=). `_, err := f()`는 "새 변수 없음"(\_ 는 변수 아님) 컴파일 에러.
  - Go 관용: 함수 하나에서 에러를 `err` 하나로 계속 돌려씀. `err2`는 냄새.
  - 규칙: 그 줄에 처음 나오는 변수 있으면 `:=`, 다 기존이면 `=`.

## [2026-07-08] 프레이밍은 계층마다 따로 — 와이어 vs 저장은 다른 봉투

- **상황:** Day 3 프로토콜 설계 — 클라가 이미 len을 붙여 보내는데 브로커가 벗기고 Append가 또 len을 붙이는 게 중복 아닌가?
- **핵심:** 같은 `[4byte len][payload]` 기법이 **두 계층에서 각자** 쓰인다.
  - **①와이어 프레이밍(네트워크):** 클라가 붙이고 handleConn이 벗김. TCP 스트림에서 payload 경계를 찾으려고.
  - **②파일 프레이밍(디스크):** `log.Append`가 내부에서 붙임. 파일에서 메시지 경계를 찾으려고.
- **배운 것:**
  - `log.Append(payload)`는 **생 payload**를 받는다. 내가 len을 미리 안 만들어 넘김 — Append가 알아서 붙임.
  - payload는 봉투 2개에 담김: 배송 봉투(와이어) → handleConn이 뜯음 → 보관 봉투(파일)에 log.Append가 다시 담음. 목적 다름, 서로 독립.
- **"그냥 통과시키면 안 되나?"의 답 (트레이드오프):**
  - 기술적으론 오늘은 가능(두 포맷이 우연히 동일).
  - 근데 안 함: ①**곧 갈라진다** — Day 4에서 PRODUCE에 파티션 키가 붙어 `[opcode][keylen][key][payload]`가 되면 통째 저장 불가(키는 라우팅용). 실제 카프카 디스크 레코드엔 timestamp·CRC 등 와이어에 없는 필드 많음. ②**계층 분리** — Append가 생 payload만 받아야 log가 네트워크를 모름. ③**아끼는 건 4byte 재인코딩(먼지), 잃는 건 두 포맷 영구 결합(큼).**
  - 결론: 싼 재인코딩으로 와이어≠저장을 독립시키는 게 정답. 통과는 "성능용 영리함"이라 Day 4에서 문다.

## [2026-07-08] 루프 변수 캡처 + 언어 버전은 go.mod가 정한다

- **상황:** 동시 Append 테스트에서 goroutine이 `offsets[i]=off` 하다 `index out of range [100]` panic
- **오해:** "Go 1.26 툴체인이니 루프 변수 캡처 버그는 안 밟는다"(내가 그렇게 말함 — 틀림)
- **원인:**
  - Go 1.21 시맨틱은 `for i`의 `i`가 **반복 전체에서 공유**. goroutine이 나중에 실행될 땐 루프가 끝나 `i==100` → 범위 초과.
  - **결정타:** 언어 시맨틱은 `go version`(툴체인)이 아니라 **`go.mod`의 `go` 줄**이 정한다. `go mod init`을 1.21 시절 돌려 `go 1.21.6`이 박혀서, 1.26 컴파일러가 **1.21 규칙**으로 컴파일함.
- **배운 것:**
  - "어느 Go로 빌드하나"보다 "**go.mod가 몇 버전을 타겟하나**"가 언어 동작을 정한다. 문제 진단 시 go.mod부터 볼 것.
  - Go 1.22+부터 반복마다 새 `i` → 버그 사라짐. go.mod를 `go 1.24`로 올려 해결.
  - 버전 무관 우회: `go func(i int){...}(i)`로 값 복사 전달. 1.22 이전 코드는 다 이 방식. 클로저는 변수를 "값"이 아니라 "참조"로 캡처함을 이해할 것.

## [2026-07-08] `-race`는 메모리 레이스만 잡는다 (논리 레이스는 못 봄)

- **상황:** 락 없이 동시 Append → 중복 offset 버그. `-race`로 돌려도 "DATA RACE" 경고 없음
- **배운 것:**
  - `-race`는 **동기화 없는 공유 메모리 접근**만 탐지. `os.File`은 내부 락으로 fd 접근을 동기화하므로 메모리 레이스가 없음.
  - 이 버그는 "두 goroutine이 같은 offset 반환"이라는 **논리적 레이스** — `-race`로 안 잡힘. **유니크성 assertion**으로만 잡힘.
  - `-race`가 조용해도 동시성이 안전한 게 아니다. 논리 불변식은 직접 테스트로 검증.
  - 레이스는 **비결정적** — 실행마다 충돌 지점 다름(중복 offset이 21→0). 재현이 들쭉날쭉한 게 정상.

## [2026-07-08] byte-offset 방식의 트레이드오프 — 경계 아닌 offset은 쓰레기를 읽는다

- **상황:** FETCH 테스트에서 레코드 중간 offset(5)을 요청하니 서버가 `EOF` 에러
- **원인:** offset이 byte 위치라, 레코드 한복판(5)에서 4바이트를 "길이"로 읽으면 payload 바이트가 길이로 해석돼 쓰레기 길이(354917) → 그만큼 읽으려다 EOF.
- **배운 것:**
  - **byte-offset의 대가:** 클라가 레코드 경계가 아닌 임의 offset을 보내면 브로커가 깨진 데이터를 읽음. 순번(0,1,2) 방식이면 브로커가 순번→위치 매핑을 쥐고 있어 이런 일이 원천 차단됨.
  - 그래서 "컨슈머는 PRODUCE/이전 FETCH가 준 offset만 쓴다"는 암묵적 계약이 생김. → **ADR "offset 표현 선택"의 트레이드오프 항목.**
  - `uint64` 최대값을 `int64`로 캐스팅하면 -1(음수) — 최상위 비트가 부호로 해석됨. 크래시는 없지만 입력 검증 없으면 이상한 에러로 샘.
- **개선 여지:** Read 에러 시 `[0x01]` 응답을 주면 클라가 "잘못된 offset"을 구분 가능(현재는 연결만 끊겨 구분 불가).

## [2026-07-09] 파티셔닝 = 전체 순서를 버리고 키 순서를 사는 거래

- **상황:** Day 4 — 단일 로그(mutex 하나)가 처리량 병목. 어떻게 뚫나
- **배운 것:**
  - **병목의 정체:** 로그 1개 = append 지점 1개 = 락 1개 → 모든 producer가 직렬화. 처리량 천장.
  - **해법:** 토픽 = 파티션 N개, 파티션 = 독립 로그(자기 파일·offset·mutex). 락이 N개로 쪼개져 병렬 append.
  - **라우팅:** `hash(key) % N`. 같은 키 → 항상 같은 파티션(결정적). 메시지 하나는 통째로 한 파티션에 감(쪼개짐 아님). 다양한 키가 여러 파일로 퍼지는 것.
  - **거래:** 전체 순서 **포기** ↔ 키 단위 순서 **유지** ↔ 처리량 N배. 현실의 순서 요구는 대부분 개체별(user별)이지 전역이 아니라, 이 거래가 남는 장사.
  - **키 정책은 producer가 정한다:** 브로커는 받은 키를 해시만. 순서 필요하면 키(user_id 등), 균형 필요하면 키 없음(라운드로빈). 핫키 편중은 키 기반의 대가.
  - **Day 3 분리의 회수:** PRODUCE에 key가 붙어 `[keylen][key][len][payload]`. 키는 라우팅용, payload만 저장 → "와이어≠저장, Append는 생 payload만" 분리가 옳았음이 증명됨.

## [2026-07-09] 셋업 비용 vs 핫패스 비용 + 경계검사 off-by-one

- **상황:** `NewTopic`의 `os.MkdirAll`이 매 메시지 부하 아니냐는 의문 / `Read` 경계검사 `>` vs `>=`
- **배운 것:**
  - **비싼 준비는 생성자에서 1회, 핫패스는 재사용.** `MkdirAll`·파일 열기는 `NewTopic`/`NewLog`(부팅 1회)에. 핫패스(`Append`)는 이미 열린 자원만 씀. → Day 2의 "파일 한 번 열어 재사용"과 같은 원칙. "파일시스템 호출을 핫패스에 넣지 마라"는 본능은 옳고, 위치(셋업이냐 핫패스냐)만 정확히 볼 것.
  - **off-by-one:** 유효 인덱스는 `0 ~ len-1`. 그래서 무효는 `< 0 || >= len`. `> len`으로 쓰면 `partition == len`이 뚫려 panic. `>`인지 `>=`인지가 정확히 한 칸을 가름.
  - **트러스트 바운더리 입력 검증:** 클라가 준 partition은 검증 필수(`Append`의 partition은 내부 생성이라 안전, `Read`는 외부 입력이라 위험). Go에서 **goroutine 내 미복구 panic은 프로세스 전체를 죽임** → 검증 안 하면 클라 하나가 브로커 전체 다운.

## [2026-07-09] 파티션 수는 유저 수가 아니라 처리량이 정한다

- **상황:** MAU 수천만 서비스면 파티션(파일)이 수천만 개 되는 것 아니냐는 의문
- **오해:** 파티션 수가 유저 수에 비례 / 파일 개수는 안 중요하고 크기만 중요
- **배운 것:**
  - **파티션 ≠ 유저.** 파티션은 작고 고정된 수(토픽당 수십~수백, 많아야 수천), **처리량·컨슈머 병렬성**으로 정함. 유저 수 무관.
  - **해싱이 분리한다:** `hash(key) % N`이 무한한 키(유저 수천만)를 N개 통에 매핑. 유저 ~10만 명이 파티션 1개 공유. "키 개수"와 "파티션 수"가 decouple됨.
  - **왜 유저당 파티션 안 만드나:** 파티션은 비싼 자원. 유저별 병렬성은 불필요, 처리량에 필요한 만큼만.
  - **파일 개수(파티션 수)엔 진짜 비용:** 열린 FD, 메모리(버퍼·인덱스·메타데이터), 리밸런싱·복구 지연, fsync 대상 증가. "파티션 너무 많으면 느려진다"는 실제 문제 → 브로커당 상한(수천). 공짜 아님, 튜닝 결정.
  - **파일 크기(파티션당)는 세그먼트+리텐션이 상한.** 하나의 거대 파일 아님, 오래된 세그먼트 삭제로 유한.
  - **결론:** 파티션 수도 크기도 유저가 아니라 **처리량·리텐션이 정하는 유한한 설계값.** loadgen 실측(초당 2.5~3.2만건, fsync 없이 = "OS 위임" 정책)도 이 맥락 — Day 6에서 fsync 정책별로 이 숫자가 어떻게 무너지는지 볼 것.

## [2026-07-09] fsync = Write는 페이지 캐시일 뿐, 디스크 보장은 별도 (Day 6 토대)

- **상황:** fsync 개념을 잘 몰랐음 → 처음부터 정리
- **배운 것:**
  - `file.Write`는 데이터를 **OS 페이지 캐시(RAM)** 에 넣고 "성공" 반환. **디스크엔 아직 없음.** OS가 나중에 게으르게 내려씀. → 정전/크래시 나면 캐시 내용 유실(성공 응답했는데 데이터 증발).
  - `fsync`(Go `file.Sync()`)는 "지금 디스크에 강제로 써라"고 **블로킹**하며 강제 → 반환 후 durable. 근데 디스크 속도라 Write(마이크로초)보다 **수백~수만 배 느림**.
  - **트레이드오프:** 매번 fsync(안전/느림) ↔ 배치 fsync(N건마다, 중간) ↔ OS 위임(빠름/유실). 내 loadgen 2.5만/초 = OS 위임 상태. Day 6 = 세 정책 처리량/p50/p99 측정.
  - **큰 그림:** 카프카 기본값은 사실상 OS 위임(매 메시지 fsync는 너무 느림). 대신 **복제로 내구성 확보** — 메시지가 브로커 3대 페이지 캐시에 있으면 한 대 죽어도 안전. "fsync 내구성 → 복제 내구성"으로 바꾼 것. → 단일 노드인 내 프로젝트는 fsync밖에 레버가 없는 한계(v2 복제로 탈출).
  - **페이지 캐시는 읽기도 빠르게:** 방금 쓴 데이터가 RAM에 있어 FETCH가 디스크 안 감. 카프카 컨슈머가 최신 데이터 빠르게 읽는 비결(+zero-copy).
  - **macOS 주의:** 맥 `fsync`는 드라이브 자체 캐시까진 강제 안 함(진짜 보장은 `F_FULLFSYNC`). Day 6 벤치마크 해석 시 감안.

## [2026-07-10] 토픽·파티션·그룹·인스턴스 계층 (여러 오해 교정)

- **상황:** Day 5 컨슈머 그룹 — 그룹/토픽/파티션 관계를 여러 번 거꾸로 이해했다 교정
- **최종 멘탈 모델:**
  - **토픽 = 이벤트 종류** (payments, orders...). 종류 다르면 다른 토픽. 프로듀서가 종류로 나눔.
  - **파티션 = 토픽 안에서 키로 나눔** (`hash(key)%N`). 순서 단위. 한 파티션 안엔 **같은 종류가 키(유저)별로 섞여** 있음.
  - **offset = 파티션 안 위치.**
  - **그룹 = 기능/목적** (analytics, payment...). 하나 or 여러 토픽 구독 가능.
  - **인스턴스 = 그룹 내 확장** (프로세스/파드). 파티션을 인스턴스끼리 나눠 가짐.
- **교정된 오해들:**
  - "로그가 기능별로 나뉘어 저장" ❌ → 저장은 **키로** 나뉨(파티션), 기능은 **읽는 컨슈머** 쪽.
  - "컨슈머가 수집해서 저장" ❌ → 넣는 건 프로듀서, 컨슈머는 **읽기만**.
  - "그룹마다 P0가 따로, 내용 다름" ❌ → **P0는 실체 하나(파일 하나)**, 모든 그룹이 **같은 P0를 읽음**. 다른 건 내용이 아니라 (1)읽고 하는 일 (2)읽은 진도(offset).
  - "한 파티션에 결제·주문 다 섞임" ❌ → 종류는 **다른 토픽**. 한 파티션 안은 같은 종류 + 키만 섞임.
  - "그룹당 프로세스 하나" ❌ → 여러 프로세스가 **같은 그룹** 공유(파티션 나눔). 그룹 ≠ 프로세스.
  - "컨슈머가 필터링" ❌ → 원하는 **토픽을 구독**해서 전체를 읽음.
- **왜 여러 그룹이 같은 파티션을 독립적으로 읽나:** append-only(소비해도 삭제 X) + 컨슈머가 offset 소유 → 팬아웃 가능. **로그 vs 큐의 결정적 차이.**
- **병렬성 상한 = 파티션 수.** 인스턴스 > 파티션이면 초과분은 놀음.
- **스코프:** MiniKafka는 단일 토픽(이름 없음). 다중 토픽은 안 만듦.

## [2026-07-10] offset 커밋 = 상태를 로그로 저장하고 replay로 복원 + at-least-once

- **상황:** Day 5 — `__offsets` 저장소 + 컨슈머 재시작 재개 구현
- **배운 것:**
  - **커밋 = "다음에 읽을 offset(재개 지점)" 저장.** 재시작 시 `GetCommitted`로 받아 거기서 이어읽기. 없으면 0.
  - **저장 방식(카프카 본질):** 커밋을 **append-only 로그로 저장**, 재시작 시 **replay로 맵 재구성**(같은 키 나중 게 이김). 카프카 `__consumer_offsets`가 정확히 이 방식 — "offset 커밋조차 로그". → 상태를 로그로 저장하고 재생해 복원하는 event-sourcing 패턴을 손으로 구현.
  - **커밋 위치 = (group, partition) 키.** 같은 파티션이라도 그룹마다 진도 독립.
  - **유실 방지 = 순서(처리 후 커밋), 빈도 아님.** 처리→커밋 = at-least-once(중복 O, 유실 X). 커밋→처리 = at-most-once(유실 O).
  - **실무:** 매 메시지 아니라 **배치/주기 커밋**(fsync랑 같은 트레이드오프) + **멱등 처리**로 재처리 무해하게. 우리 v1은 학습용이라 파티션 드레인 후 커밋.
  - **FETCH 개선:** 끝 도달 시 커넥션 닫지 말고 `[0x01][len=0]` 반환(커넥션 유지) → 컨슈머가 폴링/다음 파티션 가능. 데이터없음 ≠ 에러 구분. (Day 3부터 미룬 에러 status 응답 정리.)

## [2026-07-10] offset은 그룹별 책갈피(전역 "소비됨" 아님) + 재읽기 = seek

- **상황:** "offset 100까지 소비됐다고 저장되면 다른 컨슈머/재읽기가 막히나?" 하는 의문
- **배운 것:**
  - offset은 **전역 "소비됨" 상태가 아니라 (group, partition)별 책갈피.** `(groupA,P0)=100`과 `(groupB,P0)=50`은 완전 독립. 데이터는 삭제·표시 안 됨(append-only) → 0~100 전부 남아 누구든 읽음.
  - "컨슈머마다 offset 따로"가 아니라 **"그룹마다 따로"**. 같은 그룹 컨슈머들은 파티션을 나눠 가짐(같은 데이터 안 읽음). 같은 데이터 다른 용도 = **다른 그룹**(팬아웃).
  - **재읽기 = offset 되감기(seek).** 새 그룹 만드는 게 아님(그건 진짜 새 용도일 때만). Kafka `seek()` / `--reset-offsets`.
  - **내 구현엔 이미 있음:** `Fetch(partition, offset)`이 명시적 offset을 받음 → 컨슈머가 커밋값 무시하고 아무 offset부터 읽을 수 있음. 커밋된 offset은 "기본 재개 지점"일 뿐 제약 아님.
  - **왜 공짜:** 멍청한 브로커(시킨 offset만 서빙) + 똑똑한 컨슈머(위치 소유) + append-only(안 지움). Day 3 원칙의 회수. 로그가 재처리를 공짜로 주는 이유(큐는 불가).
  - **일회성 vs 영구:** 되감아 Fetch만 = 이번만 재읽기. 되감은 offset을 Commit = 영구 리셋(다음 재시작도 거기서).

## [2026-07-10] 복구는 두 계층 + 재개는 "커밋된" offset부터 + 무결 전제는 조건부

- **상황:** "다운 후 리플레이로 재개 = 처리하던 offset부터 이어감?" 확인하다 개념 정리
- **배운 것:**
  - **"복구/리플레이"가 두 종류 (계층 다름):**
    - ① **컨슈머 재개** — 컨슈머 죽으면 재시작 시 `GetCommitted`로 **책갈피 조회 후 그 위치부터 Fetch.** (로그 재생 아님, "저장된 지점에서 재개".)
    - ② **브로커 OffsetStore replay** — 브로커 죽으면 재시작 시 `__offsets` 로그를 **처음부터 재생해 맵 재구성.** (이게 진짜 "리플레이" — 로그로 인메모리 상태 복원.)
  - **재개 지점 = 마지막 "커밋된" offset (마지막 "처리한" 것 아님).** 커밋이 처리보다 뒤처지면(N건마다 커밋) 그 사이가 **재시작 시 중복 처리** = at-least-once. 유실 X, 중복 O → 멱등 처리로 무해화.
  - **"데이터 무결 저장" 전제는 조건부** — 실패 유형에 달림:
    - 컨슈머 다운 → 브로커·데이터 멀쩡(전제 성립).
    - 브로커 프로세스 다운(전원 살아있음) → 파일+페이지캐시 생존.
    - 전원 손실/머신 다운 + fsync·복제 없음 → 최근 tail 유실 가능.
    - → 무결 전제는 **fsync 또는 복제**가 있어야 진짜 보장. v1 단일노드의 한계, 복제(v2)로 탈출.

## [2026-07-10] Kafka vs MQ 선택 기준 — 가장 큰 축은 "순서"가 아니라 "일 큐 vs 이벤트 스트림"

- **상황:** 둘 중 선택의 가장 큰 요인이 실행 순서 보장인지 질문
- **배운 것:**
  - **가장 큰 축 = "일(task) 큐 vs 이벤트 스트림", 순서 아님.**
    - **MQ(큐):** 일 분배. "이 작업 한 번 처리하면 삭제". 경쟁 소비.
    - **Kafka(로그):** 이벤트 스트림. 유지·재읽기, 여러 그룹 팬아웃, replay 공짜.
  - **MQ만의 강점(빠뜨리기 쉬움):** **ack/nack · 재시도 · 데드레터큐(DLQ) · 우선순위 · TTL.** "작업을 반드시 완수"에 최적(이메일·결제잡·이미지처리). Kafka는 per-메시지 ack/재시도 약함(offset 기반).
  - **순서는 부차적, 오히려 Kafka가 유리:** Kafka는 키→파티션으로 **파티션 내 강한 순서**(병렬이면서). MQ는 경쟁 소비 넣으면 순서 깨짐. → "순서+병렬" 필요하면 Kafka가 편함.
  - **처리량 매우 크면 Kafka.**
  - **결정 휴리스틱:** "이 메시지를 **나중에 다른 누군가가 또 필요로 할까?**" → 예(여러 소비자·재처리·기록)면 Kafka, 아니오(이 작업 하나 처리 후 끝)면 MQ.
  - **면접 카드:** "왜 여기 Kafka 말고 MQ?" → 이 틀(일 큐 vs 이벤트 스트림 + ack/DLQ + 순서는 부차)로 답. v2 MQ 프로젝트의 "로그 vs 큐" 비교 핵심 논점.

## [2026-07-13] fsync 3정책 실측 — 내구성-처리량 트레이드오프가 숫자로

- **상황:** Day 6 — loadgen에 fsync 정책(`--fsync 0/100/1`)별 처리량·p50·p99 측정
- **실측(20000건, macOS):**

  | 정책        | 처리량   | p50   | p99   | 내구성                |
  | ----------- | -------- | ----- | ----- | --------------------- |
  | 0 (OS 위임) | 26,821/s | 33µs  | 77µs  | 최저                  |
  | 100 (배치)  | 14,835/s | 27µs  | 964µs | 중간(최대 100건 유실) |
  | 1 (매번)    | 232/s    | 4.0ms | 9.1ms | 최고                  |

- **배운 것:**
  - **매번 fsync는 OS위임보다 ~115배 느림**(26821→232). 내구성의 대가가 숫자로. 매 produce가 디스크 대기.
  - **배치(100)가 스위트스팟** — OS위임의 55% 처리량, fsync=1보다 64배 빠름, 내구성은 "최대 100건 유실"로 확보. 실무가 배치+주기 커밋 가는 이유.
  - **🎯 p50 vs p99가 왜 중요한지 실증:** fsync=100의 **p50=27µs(대부분 빠름)인데 p99=964µs(매 100번째 fsync 스파이크, 35배)**. 평균 냈으면 이 주기적 스파이크가 뭉개졌을 것. p99가 "1%의 나쁜 경험"을 정확히 드러냄. → "평균 말고 분위수" 원칙의 산 증거.
  - **macOS 캐비아:** `fsync`가 드라이브 캐시까진 강제 안 함(F_FULLFSYNC 필요) → fsync=1의 232/s는 실제 비용 **과소평가**. Linux면 더 느림.
  - loadgen은 단위 테스트와 무관 — 브로커에 TCP로 붙어 측정. fsync 값은 **브로커 `--fsync` 플래그**에서 옴(테스트의 fsyncN=0과 별개).
